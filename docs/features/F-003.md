# F-003: Portable Game State and Command Model

## Summary

Define the portable C data model for game state, player commands, game events, and result codes that later extracted rules and the native Mac app shell will share.

## Reference

- BASIC lines: `0-99`, `1000-1700`, `60000-60250`
- Screenshots: `STARTUP1.GIF`, `STARTUP2.GIF`, `TOWN.GIF`, `OUTSIDE.GIF`
- Docs: `ARCHITECTURE.md`, `BOUNDARIES.md`, `docs/modernization/porting-strategy.md`

## Scope

- Add `Core/` headers and implementation for portable game state structs.
- Model commands such as movement, entering/exiting locations, buying items, attacking, casting magic, and acknowledging prompts.
- Model emitted events without exposing AppKit, SwiftUI, Foundation, Metal, SDL, or Apple II emulator APIs from `Core/`.

## Acceptance Criteria

- `Core/` exposes a stable C API for initializing state and applying one command at a time.
- The model can represent startup, overworld, town/store, dungeon, combat, death, and quest states even if every rule is not implemented yet.
- Harness tests prove deterministic initialization and basic command/result behavior.
- `Core/` passes the boundary check from `BOUNDARIES.md`.

## Validation

```bash
rg -n "AppKit|SwiftUI|Metal|AVFoundation|Foundation|SDL|NS[A-Z][A-Za-z]+\\b|NSURL|Sandbox" Core
make -C harness test
```

## Status

Implemented. `Core/include/ak_game.h` and `Core/src/ak_game.c` define a portable state, command, result, and event model. The model intentionally reports unimplemented rule transitions as `AK_GAME_RESULT_UNSUPPORTED_RULE` so later extraction features can fill behavior without changing the command API.
